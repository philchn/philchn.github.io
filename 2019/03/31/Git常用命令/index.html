Git流程

在多人开发过程中，通常会建立一个develop分支，然后每个人去建立各自的单独分支，每个人在自己的分支上进行开发，然后合并到develop分支（如果是独自开发，觉得直接在develop进行开发也可以）。

常用命令

正常开发

（1）获取代码

git clone git地址

（2）创建本地分支

在本地创建develop分支并切换到develop分支

git checkout -b develop //也可通过git branch develop创建，不切换到develop分支

在本地创建phil分支并切换到phil分支

git checkout -b phil

推送到远端分支

git push origin phil

（3）代码提交

开发完一部分代码后，进行提交

git add .
git commit -m 'xxx' //xxx是对该次提交的描述
git push origin phil //自己的分支，没有别人提交代码，不需要考虑冲突问题，所以一般不需要pull，直接push即可

（4）合并代码到develop分支

git checkout develop
git pull origin develop //拉取dev分支上最新代码
git merge phil //合并
git push origin develop //推送到远端

（5）云构建发布（适用于阿里云那一套）

代码已合并到dev分支，当前处于dev分支上

git checkout -b daily/0.0.x //创建本地日常分支，0.0.x为版本号
git pull origin master //获取master分支上代码,防止日常发布失败
git push origin daily/0.0.x //日常分支发布（平时开发测试到此即可，真正需要发布上线的时候，才需要下面两步）
git tag publish/0.0.x //创建tag
git push origin publish/0.0.x //线上分支发布

特殊情况

（1）回滚

git log -n //显示历史版本
git reset --hard xxx //回滚到xxx版本
git push -f origin xxx //强制提交

（2）强行切换分支

git checkout  xxx -f //xxx为分支名

补充

（1）查看当前代码库的状态

git status //通常集成了git的编辑器会实时显示

（2）合并分支merge和rebase

rebase又称为衍合，是合并的另外一种选择。

由图看出，merge操作会生成一个新的节点，之前的提交分开显示。而rebase操作不会生成新的节点，是将两个分支融合成一个线性的提交

如果你想要一个干净的，没有merge commit的线性历史树，那么你应该选择rebase
如果你想保留完整的历史记录，并且想要避免重写commit history的风险，你应该选择使用merge

（3）大小写

git默认对大小写是不敏感的，如果文件最初名称是全小写，提交之后中途改为驼峰命名，会出现一些问题。

之前碰到过的是远程仓库里会同时存在两个文件，一个是原本的文件，一个是改名之后的，而本地只会有一个改名之后的文件，即使开启了对大小写的敏感，也有可能会有一些问题出现,所以在最初创建文件夹名或者文件名的时候，最好一次性定下来。

如果真的需要中途改名，可以将该文件剪切出去，提交代码（即删除了本地和远端的该文件），然后再将该文件粘贴回来，改名提交。
